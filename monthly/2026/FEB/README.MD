# Readings

## The Pragmatic Programmer (ISBN: 978-0-13-595705-9)

Chapter 9: Pragmatic Projects

- Tips in this book, which help individuals to become a better programmer, work also for teams.
- A developer who's been put into a team which doesn't care, losses enthusiasm. Teams should not tolerate broken
  windows.
- It's easy for team to get boiled: it's easy to assume someone else reacts to an incident. The solution is to
  encourage the team to monitor and maintain metrics.
- If the team is serious about learning and innovation, it should be scheduled. Doing it in the _free time_ doesn't
  work.
- The worst teams hold meeting with no structure, doesn't keep the documentation standardized and up-to-date,
  communicating with them is the nightmare. Cooperating with the best teams is a pleasure.
- Developers isolated from end users don't have context in which they work. They will not be able to make informed
  decisions.
- It's easy to get into a trap of [cargo cult](https://en.wikipedia.org/wiki/Cargo_cult). Only using superstitious
  artifacts like 'standups' or 'iteration' is not enough to make benefits of using scrum.
- It's recently popular to follow tech and methodologies of companies like Netflix. We should adapt and test parts of
  these which work in our environment.
- Overinvesting in a single solution makes us blind for alternatives.
- Manual steps leaves repeatability up to chance. Version control is required to maintain ephemeral environments. It
  should trigger builds and releases.
- Automatic testing is crucial for continuous integration. Unit tests exercise a module and are the cheapest way to
  catch small bugs before they grow up. Integration tests check if subsystems work with each other exercise contracts
  between them.
- Bug free system is not enough to be useful. Validation and verification check if they solve the real business problem.
- Performance tests help to assess if a system is ready for predicted number of users or requests, if it's scalable.
- Cause a bug deliberately and check if tests will catch it.
- The moment when a human finds a bug should be the last time a human found this bug. Automated test should be created
  to prevent it from re-occurring. Otherwise, it **will** occur again.
- Deploying to anonymous cloud requires the deployment process to be fully automated.
- Bug-free system delivered on time is not enough for users. It should help them to solve real business problems. Their
  expectations are not software-related.
- Ask stakeholders a question: how will we know a month after finishing a project if we were successful.
- Instead of _software engineers_ we should have rather been called _problem solvers_, because our primary focus should
  not be software-related.
- When taking serious responsibility for technical aspects of our systems, we should be proud of what we're doing.
- We should not defend our code from interlopers. Mutual respect is a condition for this tip to work.
- Anonymity makes room for mistakes and bad code.
- Common responsibility, supported by additional practices like pair programming, is a recommendation of eXtream
  Programming.
- My signature should be a confirmation of the best quality.
